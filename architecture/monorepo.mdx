## Overview

The repository is a **pnpm workspaces** monorepo. All packages share a single `node_modules` at the root, and pnpm's workspace protocol (`workspace:*`) wires local packages together without publishing to a registry.

```text
artifacts-monorepo/
├── artifacts/                  # Deployable applications
│   ├── api-server/             # @workspace/api-server — Express 5 API
│   └── pi-day/                 # @workspace/pi-day — React SPA
├── lib/                        # Shared libraries
│   ├── api-spec/               # @workspace/api-spec — OpenAPI spec + Orval config
│   ├── api-client-react/       # @workspace/api-client-react — generated React Query hooks
│   ├── api-zod/                # @workspace/api-zod — generated Zod schemas
│   └── db/                     # @workspace/db — Drizzle ORM + PostgreSQL
├── scripts/                    # @workspace/scripts — utility TypeScript scripts
├── pnpm-workspace.yaml         # workspace glob patterns + shared catalog
├── tsconfig.base.json          # shared TypeScript compiler options
├── tsconfig.json               # root project references (libs only)
└── package.json                # root devDeps: typescript, prettier
```

---

## Workspace configuration

`pnpm-workspace.yaml` declares four glob patterns that determine which directories are workspace packages:

```yaml
packages:
  - artifacts/*
  - lib/*
  - lib/integrations/*
  - scripts
```

The file also defines a **catalog** — a shared version registry for frequently used packages (React, TanStack Query, Vite, Tailwind CSS, Zod, and more). Individual `package.json` files reference catalog entries with `catalog:` instead of a version string, keeping version pinning centralized.

The `scripts` package is a catch-all for one-off TypeScript scripts. Each `.ts` file in `scripts/src/` has a corresponding npm script entry and can import any workspace package as a regular dependency.

## The `workspace:*` protocol

When one package depends on another in the same repo, its `package.json` uses `workspace:*` as the version range:

```json
// artifacts/api-server/package.json
{
  "dependencies": {
    "@workspace/db": "workspace:*",
    "@workspace/api-zod": "workspace:*"
  }
}
```

```json
// artifacts/pi-day/package.json
{
  "dependencies": {
    "@workspace/api-client-react": "workspace:*"
  }
}
```

pnpm resolves `workspace:*` to the local package directory at install time. No symlink gymnastics or `npm link` required — pnpm handles this natively via its virtual store.

## Dependency graph

The diagram below shows which packages consume which. Arrows point from consumer to dependency.

```text
@workspace/pi-day
    └── @workspace/api-client-react   (generated React Query hooks)

@workspace/api-server
    ├── @workspace/db                 (Drizzle client + schema)
    └── @workspace/api-zod            (Zod validation schemas)

@workspace/api-spec  ──codegen──►  @workspace/api-client-react
                     ──codegen──►  @workspace/api-zod

@workspace/scripts
    └── (any workspace package, added on demand)
```

`@workspace/api-spec` is not a runtime dependency of anything — it is a **code generation source**. Its output packages (`api-client-react` and `api-zod`) are the ones actually imported at runtime.

## Root scripts

The root `package.json` provides two top-level commands:

| Command | What it does |
|---|---|
| `pnpm run typecheck` | Runs `tsc --build --emitDeclarationOnly` for the `lib/` packages, then runs `typecheck` in each `artifacts/` package and `scripts` |
| `pnpm run build` | Runs `typecheck` first, then recursively runs `build` in every package that defines it |

<Note>
  Always run `typecheck` and `build` from the repository root, not from inside individual packages. Cross-package TypeScript project references only resolve correctly when built through the root `tsconfig.json`.
</Note>
